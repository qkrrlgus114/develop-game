# Error Popup Hell

PC 전용 브라우저 미션 게임 MVP입니다. 진짜 업무 경고와 가짜 팝업을 구분해 제한 시간 안에 미션을 끝내는 흐름을 제공합니다.

## 현재 상태
- 기준 일자: 2026-04-16
- 현재 이 워커 브랜치에는 **문서 + 시각 레이어 초안**(`src/pages/**`, `src/components/ui/**`, `src/styles/**`)만 반영되어 있습니다.
- 앱 스캐폴드(`package.json`, `vite.config.ts`, `src/main.tsx`)와 게임 엔진 연결은 다른 구현 레인과 통합이 필요합니다.
- 이 README는 `.omx/plans/**` 문서를 구현 가능한 기준선으로 정리한 문서입니다.

## MVP 목표
- 데스크톱 브라우저용 정적 웹 앱
- 핵심 흐름: 랜딩 → 플레이 방법 → 미션 선택 → 플레이 라운드 → 결과
- 최소 2개의 플레이 가능한 미션
- Vintage 비주얼 방향 + WCAG 2.2 AA 접근성 준수
- 로컬 최고 기록 저장

## 핵심 규칙
1. 승리 조건은 **모든 팝업 제거**가 아니라 **핵심 미션 완료**입니다.
2. 팝업은 `real`과 `fake`로 나뉘며, 실제 방해 요소는 올바르게 대응해야 합니다.
3. 오판은 시간 손실 또는 스트라이크로 이어집니다.
4. 시간 초과, 스트라이크 초과, 필수 실제 방해 요소 미해결 상태는 실패입니다.

## 필수 화면
| 화면 | 목적 | 핵심 요소 |
| --- | --- | --- |
| 랜딩 (`/`) | 규칙과 분위기 소개 | 핵심 카피, 게임 시작 CTA, 실패 조건 요약 |
| 플레이 방법 (`/how-to-play`) | 진짜/가짜 팝업 구분 학습 | 예시 팝업, 승/패 규칙 |
| 미션 선택 (`/missions`) | 미션 선택 | 최소 2개 미션 카드, 난이도/제한 시간 |
| 플레이 (`/run/:id`) | 핵심 게임 루프 | HUD, 타이머, 스트라이크, 카오스, 체크리스트, 팝업 레이어 |
| 결과 (`/result/:runId`) | 성공/실패 요약 | 등급, 남은 시간, 실수 내역, 다시 하기 |

## MVP 미션 기준
- 최소 2개 미션 제공
- 각 미션은 최소 3개 이상의 단계 포함
- 예시 미션 타입
  - 오피스 업무형: 메일/첨부/CC/전송 흐름
  - 개발 업무형: 충돌 해결/스테이징/푸시 흐름

## 권장 구현 구조
`.omx/plans/tech-architecture.md` 기준 권장 구조:

```text
src/
  app/                  # 엔트리, 라우팅, 앱 셸
  pages/                # 랜딩, 플레이 방법, 미션 선택, 플레이, 결과
  features/
    gameplay/           # 타이머, 카오스, 팝업 스케줄링, 판정
    missions/           # 미션 정의와 단계 처리
  components/ui/        # Vintage 공용 UI 컴포넌트
  data/missions/        # 정적 미션 데이터
  lib/                  # localStorage, 이벤트 로깅, 점수 계산
  styles/               # 토큰, 전역 스타일
```

## 디자인 가드레일
Vintage 스킬(`.codex/skills/vintage/SKILL.md`)과 UI 계획 문서를 기준으로 다음을 유지해야 합니다.
- 기본 토큰 방향: `primary=#008080`, `surface=#C0C0C0`, `text=#000000`
- 폰트 방향: `Silkscreen`, `JetBrains Mono`
- 명확한 `focus-visible` 상태 제공
- 장식보다 가독성/대비/키보드 접근성 우선
- 진짜/가짜 여부를 색상만으로 전달하지 않음

## 검증 체크리스트
`.omx/plans/test-spec-error-popup-hell.md` 기준 필수 게이트:

### 자동 검증
- 타입체크: `tsc --noEmit` 또는 동등 명령
- 테스트: 핵심 흐름(랜딩→선택→플레이→결과), 판정 로직, 점수 계산
- 린트: 수정 파일 기준 lint 통과
- 빌드: 정적 빌드 성공

### 수동 검증
- 랜딩에서 승리 조건이 즉시 이해되는지
- 미션 선택에서 최소 2개 카드가 보이는지
- 플레이 HUD(타이머/스트라이크/카오스/체크리스트)가 동시에 읽히는지
- 실제 방해 요소 대응 시 진행, 가짜 팝업 오판 시 패널티가 반영되는지
- 결과 화면에서 같은 미션 재시작/다른 미션 선택이 가능한지
- 키보드 포커스, 헤딩 구조, 대비가 접근성 기준을 만족하는지

## 알려진 구현 리스크
- 팝업 밀도가 과도하면 불공정하게 느껴질 수 있음
- Vintage 장식이 가독성을 해칠 수 있음
- 규칙 설명이 부족하면 사용자가 승리 조건을 오해할 수 있음
- 현재 저장소에는 구현 코드가 없어, 실제 명령(`npm run dev`, `npm run build`)은 스캐폴드 추가 후에만 실행 가능합니다
